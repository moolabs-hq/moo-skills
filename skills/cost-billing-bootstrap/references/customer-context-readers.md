# How the other 6 skills read from `customer-context/`

Every skill in the suite reads `.moolabs/customer-context/` before running. This reference is for skill authors: how to consume the bootstrap's outputs without re-asking the customer.

## Precondition pattern (drop into every SKILL.md)

```markdown
## Refuse-to-run preconditions

This skill reads `.moolabs/customer-context/` (generated by `/cost-billing-bootstrap`).
If the directory is absent or missing required files, refuse with:

  "customer-context/ not found at <repo>/.moolabs/customer-context/.
   Run /cost-billing-bootstrap first, OR pass --skip-context if you accept
   degraded mode (suite uses generic defaults for terminology, pricing, and
   telemetry detection)."

Required files (refuse-to-run if any missing):
  customer-context/product-summary.md
  customer-context/repo-info.yaml
  customer-context/terminology.yaml

Optional files (warn if missing; degrade gracefully):
  customer-context/pricing-model.yaml
  customer-context/telemetry-stack.yaml
```

## What each skill consumes

### `/cost-billing-discovery`

- `product-summary.md` — informs Phase 2 doc-tree synthesis (the customer's features are the leaves to look for).
- `pricing-model.yaml` — feeds Phase 4's pricing-page intersection signal (when the customer's pricing terms appear in handler names, +0.20 confidence).
- `repo-info.yaml` — skips re-running `repo_scan.py` if fresh (< 7 days old).
- `terminology.yaml` — labels usage-event candidates with the customer's terms (e.g., `generation.delivered` instead of `completion.delivered`).

### `/cost-billing-cloud-bill`

- `repo-info.yaml` — finds the cloud-SDK imports (boto3 / google-cloud-* / azure-*) to confirm clouds are actually used.
- `terminology.yaml` — labels cell ③ findings with customer terms.

### `/cost-billing-instrument`

- `repo-info.yaml` — picks the right per-language, per-framework codemod template per service.
- `telemetry-stack.yaml` — picks brownfield (extend existing spans) vs greenfield (introduce OTel) per service.
- `terminology.yaml` — emits SDK call arguments with the customer's `event_type` strings.
- `pricing-model.yaml` — informs idempotency-key derivation (if pricing uses `seat_id` for seat-billed plans, use that as the anchor).

### `/cost-billing-drift-lint`

- `repo-info.yaml` — same code-graph fusion as discovery.
- `terminology.yaml` — drift-lint reports use customer terms.

### `/cost-billing-adversarial-review`

- `terminology.yaml` — review specs use customer terms in summaries.
- `product-summary.md` — informs "does this feature actually exist?" checks (catches hallucinated features).

### `/cost-billing-reconcile`

- N/A — Skill C is engineering-internal at Moolabs; it does NOT read customer-context.

## Reading pattern (pseudocode)

```python
from pathlib import Path
import yaml

def load_customer_context(repo_root: Path) -> dict:
    ctx_dir = repo_root / ".moolabs" / "customer-context"
    if not ctx_dir.is_dir():
        raise CustomerContextMissing(ctx_dir)
    required = ["product-summary.md", "repo-info.yaml", "terminology.yaml"]
    missing = [f for f in required if not (ctx_dir / f).is_file()]
    if missing:
        raise CustomerContextMissing(ctx_dir, missing)
    return {
        "product_summary": (ctx_dir / "product-summary.md").read_text(),
        "repo_info": yaml.safe_load((ctx_dir / "repo-info.yaml").read_text()),
        "terminology": yaml.safe_load((ctx_dir / "terminology.yaml").read_text()),
        # optional:
        "pricing_model": _load_optional(ctx_dir, "pricing-model.yaml"),
        "telemetry_stack": _load_optional(ctx_dir, "telemetry-stack.yaml"),
    }
```

## Degraded mode (`--skip-context`)

If the customer hasn't run bootstrap and the integrator wants to proceed anyway:

- All required-file checks pass with warnings.
- `terminology.yaml` defaults: `end_user="end-user"`, `output="output"`, no synonyms.
- `repo-info.yaml` defaults: run `repo_scan.py` fresh every time.
- `pricing-model.yaml` defaults: empty; pricing-page intersection signal skipped.
- `telemetry-stack.yaml` defaults: detect-on-the-fly per service.

The skill emits this warning prominently:

> ⚠️ Running without customer-context. The suite is using generic defaults.
> Run /cost-billing-bootstrap to get customer-specific labeling and behavior.

## Refresh signal

When customer-context is older than 30 days AND the skill is invoked, emit:

> ℹ️ customer-context was generated <N> days ago. Consider re-running
> /cost-billing-bootstrap if product, pricing, or repo shape has changed.

Don't block; just inform.
